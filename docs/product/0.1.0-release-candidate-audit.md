## Publication Blockers

1. `Mensor` is still a working name. Trademark, package-scope, repository,
   domain, and search-identity clearance are not recorded.
2. No license decision is implemented. The repository has no `LICENSE`, while
   the package policy promises that published tarballs contain license text.
3. All three public package manifests remain `private: true` and omit final
   description, license, repository, homepage, bugs, keywords, and publication
   access metadata.
4. No security-reporting policy or private vulnerability contact is published.
5. No release workflow, trusted-publishing identity, provenance check, staged
   approval, changelog, or package migration note is implemented.
6. The README documents a source-checkout preview only. A registry installation
   command cannot be truthful before package identities and publication exist.
7. Local validation passed, but no push was requested, so remote CI for the
   candidate commits has not been observed.

These are release blockers, not reasons to add more compiler behavior. Do not
remove `private: true`, create registry packages, publish, tag, or add a release
workflow until the maintainer resolves the name, license, package scope, and
security-contact decisions together.

## Required Maintainer Decisions

| Decision | Why it cannot be inferred |
| --- | --- |
| Final product and package name | The working name has unresolved identity and availability risk. |
| Repository and package license | This creates legal permissions and patent obligations. |
| npm organization or unscoped package ownership | Availability and account ownership are external state. |
| Security contact | A public policy needs a real monitored private channel. |
| First publication mode | Trusted publisher, provenance, access level, and staged approval require account configuration. |

## Work Unlocked After Decisions

Once those decisions are explicit, one release-preparation change should update
the package manifests, add the chosen repository and package license files,
write `SECURITY.md` and release notes, add a least-privilege trusted-publishing
workflow, extend package smoke to inspect publication metadata and packed
contents, and replace the README source-preview caveat with a real installation
path. Publication, tagging, and pushing remain separate maintainer-controlled
actions.

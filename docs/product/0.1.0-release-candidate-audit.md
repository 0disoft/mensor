## Publication Blockers

1. `SECURITY.md` names GitHub private vulnerability reporting, but repository
   setting enablement and maintainer notification delivery are not yet verified.
2. The three package identities do not exist in npm yet. Their first `0.1.0`
   publication must be a manual 2FA bootstrap because npm staged publishing
   cannot create a new package.
3. The GitHub `npm-release` environment and each package's stage-only trusted
   publisher binding cannot be configured until the repository and registry
   settings are changed by the maintainer.
4. The README records the exact future installation command but correctly marks
   it unavailable until registry publication is observed.
5. Local validation passed after these release-control changes, but no push was
   requested, so remote CI for the complete candidate has not been observed.
6. No npm publication, staged approval, fresh registry-consumer smoke, Git tag,
   or GitHub release exists yet.

These are external release actions, not reasons to add more compiler behavior.
Do not publish, approve a staged release, tag, or create a GitHub release until
the current candidate passes the complete local and remote gates.

## Resolved Publication Decision

| Decision | Resolution |

| Later publication | Configure one stage-only trusted publisher per package and dispatch `.github/workflows/release.yml` manually |
| Token policy | Do not create or store a long-lived npm automation token |
| Approval policy | npm staged approval remains a maintainer action; no external maintainer is required |

## Remaining Maintainer Runbook

Follow `docs/releasing/runbook.md`: enable and test private vulnerability
reporting, push and observe remote CI, run the one-time package bootstrap,
configure the GitHub environment and three trusted publishers, verify a fresh
registry consumer, then create the matching tag and GitHub release. Publication,
tagging, and pushing remain separate maintainer-controlled actions.

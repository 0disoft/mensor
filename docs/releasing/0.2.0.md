# Migrating to 0.2.0

Version `0.2.0` adds an opt-in machine-readable inspection declaration. The
default output remains DiagnosticReport v1, so existing consumers can upgrade
without changing snapshots or parsers.

## Packages

Upgrade the fixed package set together:

- `@0disoft/mensor-contract@0.2.0`
- `@0disoft/mensor-compiler@0.2.0`
- `@0disoft/mensor-cli@0.2.0`

## CLI

Existing automation remains unchanged:

```text
pnpm exec mensor check . --json
```

Select Check Output v2 explicitly when the consumer is ready to inspect domain
states:

```text
pnpm exec mensor check . --json --report-version 2
```

`--report-version` accepts only `1` or `2` and requires `--json`. Omitting it
continues to emit revision 1. Exit statuses remain `0`, `1`, `2`, and `3` with
their existing meanings.

## Contract Package

Revision-aware consumers may import:

```ts
import {
  parseCheckOutputV2,
  parseDiagnosticReportV2,
  type CheckOutputV2,
  type DiagnosticReportV1,
  type DiagnosticReportV2,
} from "@0disoft/mensor-contract";
```

The schema is exported as
`@0disoft/mensor-contract/schemas/check-output-v2.schema.json`. The existing
`DiagnosticReport` and `parseDiagnosticReport` names remain revision-1
contracts.

## Compiler Package

The existing call retains its revision-1 result type:

```ts
const result = await checkProject({ root });
```

An explicit literal selects the revision-2 overload:

```ts
const result = await checkProject({ root, reportVersion: 2 });
```

Inspection state is compiler-derived. Project configuration cannot supply or
override it. `checked` means the applicable configured rules completed; the
diagnostic report still owns pass or fail. Missing optional configuration is
`not-configured`, while runtime semantics are always `out-of-scope`.

## Rollback

Remove `--report-version 2` or `reportVersion: 2` to return to revision-1 output
without downgrading the packages. Do not rewrite revision-2 output into a v1
shape by dropping `inspection`; use the maintained v1 producer path instead.
